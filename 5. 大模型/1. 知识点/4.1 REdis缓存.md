# REdis缓存
一种基于**内存**的高性能**键值（Key-Value）存储系统**，核心作用是作为应用与数据库之间的**高速缓存层**，暂存频繁访问的**热点数据**，以极致速度响应请求，大幅降低数据库压力、提升系统吞吐量与响应速度。
# 工作流程
Redis 缓存的核心是**内存换速度**，标准读写流程如下：
1. **读流程（Cache Hit/Miss）**
    - 应用查询数据时，**优先查 Redis**。
    - **命中（Hit）**：数据存在，直接内存读取返回（微秒级）。
    - **未命中（Miss）**：查询后端数据库（MySQL 等），查到后**回写 Redis**并设置过期时间，下次直接读缓存。
2. **写流程（数据一致性）**
    - 通常是**先更新数据库，再删除 / 更新 Redis 缓存**（或延迟双删），保证数据最终一致。
    - 高一致性场景需结合消息队列、分布式锁等机制。
# Redis 的核心特性
**高性能**：数据存储在内存中，读写速度极快。
**持久化**：支持RDF和AOF两种持久化。
**灵活性**：支持多种数据和丰富指令。
**简单易用**：提供直观的API，易于集成。
**集群与高可用**：支持主从复制、哨兵、Cluster 模式。
## 持久化机制
### RDB（快照）
按照指定时间间隔将内存数据以二进制快照写进`.rdb`文件。
**优点**：文件紧凑、恢复速度快
**缺点**：可能丢失最后一次快照后的数据
### AOF（追加日志）
记录每条写命令，以append方式追加到`.aof`文件。
三种fsync策略：

| 策略         | 说明           | 安全性   | 性能  |
| ---------- | ------------ | ----- | --- |
| `always`   | 每次写操作都 fsync | 最高    | 最低  |
| `everysec` | 每秒 fsync（默认） | 最多丢1秒 | 均衡  |
| `no`       | 由 OS 决定      | 最低    | 最高  |
## 内存管理与淘汰策略
当内存达到 `maxmemory` 上限时，触发淘汰：

| 策略               | 说明                        |
| ---------------- | ------------------------- |
| `noeviction`     | 不淘汰，写操作返回错误（默认）           |
| `allkeys-lru`    | 对所有 key 使用 LRU 算法         |
| `volatile-lru`   | 只对设置了 TTL 的 key 使用 LRU    |
| `allkeys-lfu`    | 使用 LFU（最少频率使用，Redis 4.0+） |
| `volatile-ttl`   | 淘汰 TTL 最短的 key            |
| `allkeys-random` | 随机淘汰                      |
