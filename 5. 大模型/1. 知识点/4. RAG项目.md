 # 项目目标
**RAG思想**：结合检索增强（RAG），用外部知识弥补模型固有知识边界。
该系统基于RAG架构，能够通过实时检索相关知识库中的信息来增强大模型的生成能力，确保回答的准确性和时效性。与此同时，系统通过自动化处理学生的答疑需求，极大地减轻了人工客服的工作压力，从而实现了高效、低成本的知识服务。
# 项目周期
1. 技术选型
2. 技术架构
3. 数据库设计
4. 分模块代码实现
	- 第一步：将现有后台搜集的FQA数据集存储到Mysql数据库中
	- 第二步：基于query实现Mysql数据库检索：将query和现有问题匹配（做相似度计算），如果阈值>=0.85，就认为问题比较明确，直接返回对应的答案；否则，进入RAG检索系统
	- 第三步：搭建本地知识库：对本地文档加载读取；进行文档分割；文档向量化；存储向量数据库（Milvus）
	- 第四步：基于query实现Milvus数据库检索：将query进行向量表示，并从Milvus数据库中检索出相似的top-k个文本段。
	- 第五步：将query和检索出的top-k文本段拼接，送入大模型，实现预测。
5. 测试
6. 回调
7. 项目交付

## Redis缓存：
### 语义缓存：
把每次问题的 Embedding 向量也存入 Redis，新问题进来时先计算其向量与缓存中历史问题向量的余弦相似度，超过阈值（比如 0.92）就认为是"同一个问题"，直接返回缓存答案，整个过程不到 10ms，完全跳过向量检索和 LLM 调用。
### 会话历史存储
多轮对话场景下，每次提问都需要携带历史上下文才能让 LLM 理解"它"指的是什么。Redis 的列表结构天然适合存储这种滑动窗口式的对话记录。
### Embedding 向量缓存
向量化本身也有计算成本，相同的文本没必要反复推理。Redis 可以缓存已经计算过的 Embedding，对高频词汇和固定文档块效果显著。
### 限流与请求队列
LLM API 有并发上限，高峰期直接打过去容易触发限流报错。Redis 的原子计数器和列表可以做令牌桶限流和请求排队：
## 基于mysql数据库的问答
```
用户问题
  ↓
BM25 全文检索 → 取 Top-50 候选（关键词粗筛）
  ↓
Softmax 归一化 → BM25 分数转概率，筛掉明显不相关的候选
  ↓
KNN 向量精排 → 对剩余候选计算余弦相似度，加权融合得到最终分
  ↓
阈值判断 → 超过阈值直接返回答案，否则走 RAG
```
一个**结构化 QA 知识库 + 两阶段相似度检索**的机制，本质上是用传统数据库技术模拟"记忆检索"。
### Mysql数据库
MySQL 里存的不是普通文本，而是一张结构化的 QA 表，每条记录包含问题、答案、向量、元数据四类字段
### BM25粗筛
通过BM25算法进行粗筛，BM25 是第一道门，目标是从数万条 QA 记录中快速缩小候选范围，速度是关键，精度要求不高。获取相应BM25得分
### softmax转换
通过softmax将Bm25算法得分分布到0~1之间，
Softmax 在这里是一个**分数标准化 + 差异放大器**——它不改变候选的排名顺序，但把原始分数转换成有意义的概率值，把 BM25 无界的原始分数归一化，方便和 KNN 分数做加权融合，并且通过指数放大让最优候选更突出，减少"勉强命中"的情况。筛选去除不相关候选。
### KNN分数加权
KNN 加权再用这些权重对 K 个近邻的得分做加权平均，最终得出一个更准确反映"这个问题和历史 QA 库有多匹配"的综合分数
### 阈值判断
超过阈值直接返回答案，否则走 RAG

## 大模型监督分类器的实现原理
**用大模型生成的标注数据来监督训练一个轻量分类模型**。
核心思路是"知识蒸馏"的变体：用 GPT-4 这样的大模型生成高质量训练标签，再用这些标签训练一个轻量模型（如 BERT），推理时只跑轻量模型，大模型只在数据准备阶段出现。

## 路由机制
分类器判断问题类型后，选择其中**一种**最匹配的策略执行，而不是四种同时跑。
### 实体检索
适用于问题中包含明确实体名称（专有名词、术语、定义）的情况，直接用原始问题做向量检索，匹配最精准的词条说明。
### 知识问答检索 (HyDE)
HyDE（Hypothetical Document Embeddings）的核心思路：用户问题的向量和知识库文档的向量往往在空间中不对齐。先让 LLM 根据问题生成一段"假设性答案文档"，用这段文本的向量去检索，命中率会大幅提升。
### 子查询检索 — 拆分问题并行检索再融合
当问题包含多个独立维度时，单一向量检索无法同时覆盖所有角度。大模型先将问题分解为若干子问题，每个子问题独立检索，最终将多路结果汇总送给 LLM 综合回答。
### 迭代问答检索 — 多轮追问，逐步逼近答案
适用于需要链式推理的深度问题。第一轮检索的结果可能不足以直接回答，LLM 基于初步结果提炼出新的子问题，再发起第二轮检索，如此迭代直到信息充分。
### 风险
分类器本身的准确率决定了整个系统的上限。如果大模型误判了问题类型，就会用错检索策略，导致召回结果质量下降。
# milvus数据库模块
Milvus 是**开源向量数据库**，**存向量 → 快速查最相似的向量**
父块（Parent Document / 父分片）：**存储完整的原始文档数据**
子块（Child Chunk / 子分片）：**专门用来做向量检索**
通过BGE-M3模型将子块转换为，稀疏向量和稠密向量。

| 向量类型     | 本质                          | 核心作用                   | 匹配逻辑                     |
| -------- | --------------------------- | ---------------------- | ------------------------ |
| **稀疏向量** | 词频 / 权重构成的高维向量，每个维度对应一个具体的词 | **字面 / 关键词匹配**，保证「不遗漏」 | 靠**关键词重合度**打分，完全不懂语义     |
| **稠密向量** | 大模型生成的低维向量，整体代表文本语义         | **语义 / 深层匹配**，保证「更精准」  | 靠**向量空间相似度**打分，理解上下文和同义词 |
- **稀疏向量**：擅长**精确匹配专业术语、关键词、专有名词**，靠字面重合度召回内容，解决「漏检」问题。
- **稠密向量**：擅长**深度匹配语义、上下文、具体内容**，靠语义相似度召回内容，解决「错检」问题。
