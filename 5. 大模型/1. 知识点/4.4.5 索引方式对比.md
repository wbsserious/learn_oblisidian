# 索引方式选择依据：
1. 数据量是第一约束
2. 响应时间决定搜索策略
3. 召回率与延迟的本质权衡
4. 写入频率：在高写入场景中会成为瓶颈
## 数据量是第一约束
数据量直接决定哪些索引"物理上可行"：
```
< 100万    所有索引均可，FLAT 都能在 500ms 内完成
100万~1千万  FLAT 退出，HNSW / IVF_FLAT 主导
1千万~1亿   内存压力开始显现，IVF_SQ8 / IVF_PQ 进入视野
1亿~10亿   内存成为硬约束，IVF_PQ / DiskANN 是仅有的选项
> 10亿     DiskANN 或分布式 IVF_PQ，单机 HNSW 不可能
```
## 响应时间决定搜索策略
```
< 10ms    只有 HNSW（内存图索引）能稳定达到
          IVF 系列因聚类扫描开销难以做到
< 50ms    HNSW、IVF_FLAT、IVF_SQ8 均可
          DiskANN 在 NVMe 上也能勉强达到
< 200ms   IVF_PQ、DiskANN 均可接受
< 1s      大数据量下 DiskANN 的正常水平
离线批处理  FLAT 也可用，不需要低延迟
```
## 召回率与延迟的本质权衡
同一索引类型内，召回率和延迟是一对不可兼得的参数：
```
索引       控制参数     召回率 95% 时的典型延迟
HNSW       ef           3~8ms（百万级）
IVF_FLAT   nprobe       10~30ms
IVF_SQ8    nprobe       8~25ms
IVF_PQ     nprobe       5~20ms（精度换延迟）
DiskANN    search_list  15~50ms（十亿级）
```
## 一个经常被忽略的维度：写入频率
索引的更新代价差异巨大，在高写入场景中会成为瓶颈：
```
FLAT      任意时刻插入，无额外代价，Growing Segment 直接追加
HNSW      插入新节点需要建图，但 Milvus 异步完成，在线影响小
          删除是软删除，Compaction 后图质量可能下降
IVF 系列  nlist 聚类中心是离线训练的，新插入数据直接归入最近簇
          不需要重新训练，写入代价低
DiskANN   每次写入后图结构需要更新，高频写入场景性能较差
          更适合批量构建后只读或低频更新的场景
```
如果写入 QPS 超过 10万/秒，HNSW 和 DiskANN 需要搭配足够多的 DataNode，并接受索引构建有一定滞后。

# 典型场景速查

| 场景      | 数据量      | 延迟要求    | 推荐索引             | 关键理由        |
| ------- | -------- | ------- | ---------------- | ----------- |
| RAG 知识库 | 100万~1千万 | < 100ms | HNSW             | 召回率直接影响答案质量 |
| 电商实时推荐  | 1千万~1亿   | < 20ms  | HNSW + replica=2 | 低延迟、高并发     |
| 图像搜索    | 1亿+      | < 200ms | IVF_PQ 或 DiskANN | 内存有限，规模大    |
| 文档语义检索  | 1千万      | < 50ms  | Dense+Sparse 混合  | 兼顾语义和关键词    |
| 离线相似度计算 | 任意       | 无要求     | FLAT 或 IVF_PQ    | 精度/吞吐优先     |
| 金融风控    | < 100万   | < 10ms  | FLAT 或 HNSW      | 精确度第一       |
| 安防人脸识别  | 1千万~1亿   | < 50ms  | IVF_FLAT         | 精度和速度均衡     |
| 基因序列搜索  | 亿级       | 分钟级     | DiskANN          | 超大规模，延迟宽松   |
