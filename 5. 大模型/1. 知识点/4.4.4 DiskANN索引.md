# DiskANN 诞生的背景
在 HNSW 之后，向量索引面临一个工程瓶颈：**内存墙**。
HNSW 要求整个图结构和原始向量常驻内存，对于十亿级数据：
$$内存需求=10^9×768×4B≈2.86 TB$$
传统的解决方案是分布式部署或 IVF_PQ 压缩，但前者运维复杂，后者以牺牲精度为代价。
# DiskANN 核心思想
**SSD 的随机读延迟已降至 100μs 量级，完全可以承载向量搜索的 IO 访问模式**。、
# DiskANN 核心数据结构：Vamana 图
DiskANN 的图索引称为 **Vamana 图**，它在 HNSW 的基础上做了针对磁盘访问的重新设计。
HNSW 是多层图，用层次结构减少搜索跳数。Vamana 是**单层图**，通过更激进的长程边（long-range edges）直接在一张图上实现快速导航。
单层图的优势是：磁盘访问模式更规律，每次读取一个节点的所有邻居可以在一次 IO 中完成（将节点数据对齐到扇区大小）。
## Vamana 图的构建算法
**第一步：初始化随机图**
每个节点随机连接$R$个邻居，是最大出度（类比 HNSW 的$M$）。
**第二步：迭代优化——贪心搜索 + 鲁棒剪枝**
对每个节点$p$，执行如下操作：
```
1. 从中心点 s 出发，贪心搜索找到距 p 最近的 L 个候选节点集合 V
2. 将 p 的当前邻居也加入 V（候选池扩充）
3. 用 RobustPrune(p, V, α, R) 为 p 选出新的邻居集合
4. 对每个新邻居 q：若 q 的出度未超 R，添加反向边 q→p；
   否则对 q 也执行 RobustPrune
```
RobustPrune 算法（核心代码）
```python
def RobustPrune(p, candidates, α, R):
    """
    p: 当前节点
    candidates: 候选邻居集合（已按距p距离排序）
    α: 松弛因子，α ≥ 1
    R: 最大出度
    """
    result = []
    while candidates and len(result) < R:
        # 取最近的候选
        p_star = candidates.pop_min_dist_to_p()
        result.append(p_star)

        # 关键剪枝：移除被 p_star "遮挡" 的候选
        # 若某候选 r 满足 α·dist(p_star, r) ≤ dist(p, r)
        # 则 r 可以通过 p_star 中转到达，不需要直接连边
        candidates = [
            r for r in candidates
            if α * dist(p_star, r) > dist(p, r)
        ]
    return result
```
**α 参数的几何意义：**
$\alpha = 1$ 时，任何可以被中转覆盖的边都删除，图最稀疏;$\alpha > 1$ 时，只有"中转路径显著更短"才剪枝，保留更多冗余边，图更稠密但导航性更强。论文建议第一遍$\alpha=1$，第二遍$\alpha=1.2$，两遍构建达到最优图质量。
这个剪枝策略与 HNSW 的启发式邻居选择本质相同，但 DiskANN 更明确地用$\alpha α$ 控制"遮挡容忍度"。
# 磁盘布局设计
这是 DiskANN 工程实现的精髓，直接决定 IO 效率。

每个节点在磁盘上占据固定大小的扇区块：

```
┌─────────────────────────────────────────────┐
│  原始向量（d × 4 字节，float32）             │  ← 距离计算用
├─────────────────────────────────────────────┤
│  邻居列表（R × 4 字节，uint32 节点ID）       │  ← 图导航用
├─────────────────────────────────────────────┤
│  填充字节（对齐到扇区边界，通常 4096 字节）   │  ← IO 对齐
└─────────────────────────────────────────────┘

每节点磁盘占用 = ceil((d×4 + R×4) / 4096) × 4096 字节
```

这种布局使得读取一个节点只需**一次 SSD 随机读**，既取到了向量（用于距离计算），又取到了邻居列表（用于图导航），无需二次 IO。
# 内存中的压缩向量（PQ 码）
DiskANN 在内存中存储所有向量的 **PQ 压缩版本**，用于搜索时的快速距离估计：
```
内存布局：
┌────────────────────────────────────┐
│  节点 0 的 PQ 码（m 字节）          │
│  节点 1 的 PQ 码（m 字节）          │
│  ...                               │
│  节点 N-1 的 PQ 码（m 字节）        │
└────────────────────────────────────┘
内存占用 = N × m 字节（远小于原始向量）
```

以 $N=10^9, d=128, m=32$  为例：
- 原始向量：$10^9 \times 128 \times 4 = 512$ GB
- PQ 压缩码：$10^9 \times 32 = 32$ GB（压缩比 16 :1）

## 完整查询算法
```python
def diskann_search(query, K, search_list, beam_width):
    """
    search_list: 候选集大小（类比 HNSW 的 ef），越大召回越高
    beam_width:  每轮并行 IO 的节点数（影响吞吐与延迟）
    """
    # ── 阶段一：内存初始化 ──────────────────────────────────
    # 预计算 PQ 距离查找表：query 与每个子空间所有码字的距离
    # 形状：[m 个子空间, 2^nbits 个码字]
    lut = build_pq_lookup_table(query)      # O(m × 2^nbits × d/m) ≈ O(d × 256)
    visited   = set()
    beam      = [center_node]               # 从图的中心点出发
    results   = MaxHeap(capacity=search_list)
    # ── 阶段二：Beam Search 主循环 ──────────────────────────
    while beam:
        # Step A：对 beam 中节点发起批量并行 SSD 读
        # 每个节点读取：原始向量（d×4B）+ 邻居列表（R×4B）
        node_data = ssd_parallel_read(beam)       # W 次并行 IO
        for node in node_data:
            # Step B：用原始向量计算精确距离（重排）
            exact_dist = dist(query, node.vector)
            # Step C：更新结果集
            results.push((exact_dist, node.id))
            # Step D：将未访问邻居加入候选
            for neighbor_id in node.neighbors:
                if neighbor_id not in visited:
                    # 用 PQ 查找表快速估计距离（无需磁盘读）
                    pq_dist = pq_distance(lut, pq_codes[neighbor_id])
                    beam_candidates.push((pq_dist, neighbor_id))
                    visited.add(neighbor_id)
        # Step E：从候选中选 beam_width 个最优节点进入下一轮
        # 只选 results 中最差结果 更近的候选（剪枝条件）
        beam = select_top_beam(beam_candidates, beam_width, results)
    return results.top_K(K)
```

**两次距离计算的分工：**
PQ 距离（内存，快）用于决定"要不要去读这个节点"，精确距离（SSD 读后，准）用于决定"这个节点是否进入最终结果"。前者是导航工具，后者是评判标准，两者缺一不可。
